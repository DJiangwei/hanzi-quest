## What this is

**hanzi-quest** (deployed as `hanzi-adventure.vercel.app`) — a personal Chinese-character learning game for the maintainer's 6-year-old daughter Yinuo. Pirate-adventure art direction. Parent (David) authors weekly character lists in an admin panel; AI generates kid-friendly scenes; child plays a 12-15-level "island" of mini-games per week, with a boss kraken + treasure-chest gacha on weeks with ≥10 characters.

It is **not** a multi-tenant SaaS. Optimize for Yinuo's daily fun, not for theoretical other users.

## Landmines (things that have bitten us)

- **Shared-pack week access**: rows in `weeks` with `parent_user_id IS NULL` belong to the curriculum pack, not a parent. Use `getPlayableWeekForChild(childId, weekId)` for any access check — NOT `getWeekOwnedBy(weekId, parentId)`. The latter silently 404s for pack weeks.
- **`'use server'` files** (anything in `src/lib/actions/`) can only export async functions. If you need to export an error class or sync helper, put it in `src/lib/errors/` (pure, client-safe) — do NOT import from `@/lib/db/*` in client code, postgres pulls in `fs`/`net`/`tls`.
- **`ZodiacIconDefs`** must be mounted in an ancestor of any `<ZodiacIcon>` use. Currently mounted in `src/app/play/[childId]/layout.tsx`.
- **React 19 strict mode** double-renders. Any `onComplete?.()` in a render body double-fires — wrap in `useEffect`.

- **All card grants surface through one `CardChestReveal` (card-reveal-polish).** `SceneRunner` holds a `revealCards: RevealCard[]` queue and renders `<CardChestReveal>` (tap-to-open chest, sequential) in BOTH the done/fanfare branch (boss + perfect-week cards) AND the mid-scene branch (weekly 大礼包 cards) — the two branches are mutually exclusive (`done ? fanfare : scene`), so the shared queue shows in whichever is active. Grant results (`CardGrantResult`/`GiftCard`) carry `slug/nameZh/nameEn/loreZh/loreEn`; the reveal resolves the glyph via `getPackMeta(packSlug)` (zodiac pack → `ZodiacIcon` by passing `emoji=null`; else `resolveRevealEmoji?.(slug) ?? themeEmoji`). `GiftPackReveal` + the `LevelFanfare` card banner are retired. Adding a new card source = produce `RevealCard[]` and enqueue it.
- **Lianliankan path-finding endpoints are tiles, not empty.** PR #57's `findPath` BFS allows `from` and `to` to contain tiles (they're the matchees) but every intermediate cell MUST be empty. Common refactoring bug: tightening the "must be empty" check to include endpoints, which breaks every match. Test cases `'treats endpoints as tiles'` and `'returns null when intermediate cell has any tile'` guard against this — don't relax their phrasing without thinking. BFS state key is `(row, col, dir, bends)` on all 4 components; visited set keyed on the full tuple so a cell can be revisited from a different direction or with fewer bends already used.
- **`word_match` template stays in DB after PR #57.** Same retirement pattern as `pinyin_pick` (PR #35) and `visual_pick` (PR #51). `scripts/retire-word-match.ts` flips `scene_templates.is_active=false`; `compile-week.ts` no longer emits `word_match` in sight slot 1 (lianliankan replaced it); but `WordMatchScene.tsx`, `WordMatchConfigSchema`, and `SceneRunner case 'word_match'` ALL stay for backwards-compat with old `scene_attempts` rows that reference the template. If you ever re-enable, flip `is_active=true` AND re-add slot logic.
- **Lianliankan board has a 1-cell empty border by design.** The inner 4×2 (8 cells = 4 pairs) is wrapped by an outer ring of empty cells so paths can route around remaining tiles. Don't "optimize" by dropping the border — most Z-bend paths become impossible and the game devolves into trivial straight-line matching. The border is fundamental to the puzzle.
- **Lianliankan blocked-path matches deselect with shake; they do NOT trigger reshuffle.** PR #57 spec §4.3 was explicit: matching pair with no valid ≤2-bend path → shake + deselect, NO penalty, NO reshuffle. Reshuffle is reserved for the actual deadlock case (no valid pair-with-path remaining anywhere) handled by a separate `useEffect`. The spec reviewer caught an implementer who initially deviated — silent board mutation on tap is confusing for a 6yo and removes the puzzle information the kid had built up. Keep the two cases distinct.
- **`AVATAR_SLOT_IDS` array order IS the SVG render order.** PR #58 made this explicit by replacing the hardcoded 4-layer composition in `AvatarRender` with an iteration over the array. Back → front layering: background → decor → head → pants → top → hair → hat. Reordering breaks anatomical layering (e.g. swapping hair before head would draw hair under the face). Adding a new slot = single-array edit + a new SVG component per item — no other code change required.
- **`decor` slot has no default.** `DEFAULT_AVATAR` deliberately omits the `decor` key (it's optional/expressive). Code that iterates equipped items must handle missing keys gracefully — the `AvatarRender` slot iteration checks `if (!ref) return null` for exactly this case. Don't backfill a default decor item without re-evaluating the design; the empty state is intentional.
- **Avatar `theme` is `text` not a pgEnum.** PR #58 added the column as plain text so future theme additions (Indian Ocean, Space, Exploration) require zero migration. Validate at the action layer via `isAvatarTheme(value)` from `src/lib/avatar/themes.ts` if any user-controlled input touches the field. The TypeScript union `AvatarTheme` is the single source of truth in code — never write a theme literal that isn't in that union.
- **Avatar item seed script must set `theme` on every new item.** PR #58's `scripts/migrate.ts` backfill pass sets `theme = 'pirate' WHERE theme IS NULL`. If you add a new non-pirate item but forget to seed its theme value, it'll be NULL → backfilled to 'pirate' on next deploy → wrong theme. Always set the theme in the catalog (`ItemDef.theme`) AND verify it flows through the seed insert.
- **Card grant cap is DAILY now, not weekly (card-economy-v2).** `pullCardInTx` reads/writes `child_card_grants_daily(childId, dayUtc, count)` and caps at `DAILY_CARD_CAP = 10`. The old `child_card_grants_weekly` table + `WEEKLY_CARD_CAP` constant are **dead but retained** (append-only). Do NOT re-point grant code at the weekly table. Callers pass `dayUtc = todayUtcIso()` (NOT `mondayOfIsoWeek(...)`). Result fields are `cardsToday` / reason `daily_cap_reached` (renamed from `cardsThisWeek` / `weekly_cap_reached`).
- **Boss creatures are derived from `weeks.weekNumber`, registry is client-only.** `getBossCreature(weekNumber)` in `src/lib/scenes/boss-roster.ts` returns `BOSS_ROSTER[(weekNumber-1) % 10]`. The registry holds React components — `BossScene` calls it itself; NEVER pass a roster entry across an RSC boundary (same hazard as `packRegistry`). Adding an 11th creature shifts which boss appears for weekNumber>10 (`% length` wraps). To add a creature: drop a `bosses/<Name>.tsx` implementing `BossCreatureProps` (4 states `intro/idle/damage/defeat`, reduced-motion static poses — copy `bosses/Kraken.tsx`) + append a `BOSS_ROSTER` entry. The parametrized smoke test `tests/unit/boss-creatures.test.tsx` auto-covers any new entry. No DB column.
- **`BossScene` victory is delayed by `DEFEAT_MS` (1200ms).** Post boss-animations, the winning answer enters a `defeating` phase that plays the creature's defeat animation, THEN a `setTimeout` fires `onComplete(true)`. Do NOT restore a synchronous `onComplete` on the final correct answer — it would skip the defeat animation. The kid-lost screen shows the creature in `idle`. `reset()` (retry) sets phase back to `intro` so the intro effect re-arms.
- **Boss reward is the AUTO card grant, not a chest button.** Since PR #52 the boss card is granted automatically by `pullCardForChild('boss_clear', sessionId)` in `finishLevelAction` and shown via `LevelFanfare`'s `cardGrant` banner. The legacy manual "开启宝箱" button + `pullFreeFromBoss` (old zodiac-only free pull keyed on `weekProgress.freePullClaimed`) was removed in PR #64 because it double-rewarded and threw "开宝箱失败". `pullFreeFromBoss` + `TreasureChestReveal` still exist (unused) — the post-reveal-card-polish PR will wire `TreasureChestReveal` to reveal the *already-granted* card (no server pull). Don't re-add a button that calls `pullFreeFromBoss`.
- **The weekly check-in 大礼包 deliberately bypasses the daily cap.** `grantGiftPackInTx` (in `grants.ts`) NEVER reads or writes `child_card_grants_daily` — that's the whole point ("on top of" the daily cap). It grants **one card per active pack** (count derived from active `collection_packs`, not a constant — a future 6th pack auto-joins the gift). Idempotent **once per UTC week** via `cardGrantsLog(childId, 'weekly_checkin', weekStartUtc)` (23505 → `already_granted`). If you add a new card source, decide consciously whether it should consume the daily cap (`pullCardInTx`) or bypass it (gift-pack pattern).
- **Weekly gift trigger rides the daily-login signal.** `claimWeeklyGiftIfDue(childId)` is called from `finishAttemptAction` ONLY inside the `if (daily.awarded)` branch (a fresh UTC-day check-in). It counts distinct check-in days this week via `getActivityForRange`'s `dailyLoginBonus` flag and fires at `≥ WEEKLY_CHECKIN_THRESHOLD (5)` — using `≥` not `== 5` so a child already past 5 mid-week still gets it (the `cardGrantsLog` guard keeps it once-only). It's a **trust-caller** (no `requireChild`, like `pullCardForChild`) because its caller is already auth-gated. The gift surfaces via `finishAttemptAction`'s `giftPack` return field → `SceneRunner` → `GiftPackReveal` (NOT the boss/`finishLevelAction` path — the gift fires on ordinary scene attempts).
- **Adding an avatar slot? Update the `SLOTS` array in `scripts/seed-shop-avatar-items.ts` too.** PR #58 expanded `AVATAR_SLOT_IDS` 4 → 7 (added hair/pants/decor) but the seed script's hardcoded `SLOTS` array still upserted only the original 4 `avatar_slots` rows. Result: the post-merge seed run threw `FK violation: avatar_items.slot_id='hair' not present in avatar_slots` — the four-green gate missed it because tests mock `@/db`, so this class of bug ONLY surfaces against a real DB (fixed in the PR #59 hotfix). When adding a slot, edit BOTH `AVATAR_SLOT_IDS` (defaultLook.ts) AND the `SLOTS` array in the seed script; keep `displayOrder` aligned with render order. Same hazard applies to any seed script with a hardcoded enum/id list that mirrors a TS source-of-truth array.
