## What this is

**hanzi-quest** (deployed as `hanzi-adventure.vercel.app`) — a personal Chinese-character learning game for the maintainer's 6-year-old daughter Yinuo. Pirate-adventure art direction. Parent (David) authors weekly character lists in an admin panel; AI generates kid-friendly scenes; child plays a 12-15-level "island" of mini-games per week, with a boss kraken + treasure-chest gacha on weeks with ≥10 characters.

It is **not** a multi-tenant SaaS. Optimize for Yinuo's daily fun, not for theoretical other users.

## Hard rules (non-negotiable)

1. **Never re-litigate locked decisions** without strong reason. Locked: pirate-adventure art direction; DeepSeek V4 Pro as AI provider; shared-pack model (rows in `weeks` table with nullable `parent_user_id`); pinyin hidden by default; age-appropriate vocab. If you want to change one, say so explicitly and explain why.
2. **`pnpm typecheck && pnpm lint && pnpm test && pnpm build` must all be green** at PR open.
3. **Drizzle migrations are append-only.** Never edit a committed `drizzle/*.sql`. Generate a new one. Schema source of truth is `src/db/schema/*.ts` — SQL is generated from it, not vice versa.
4. **Tests mock external boundaries** (`@/db`, `@clerk/nextjs/server`, `next/cache`, `next/navigation`, `ai`). No test hits a real DB or network.
5. **Scripts that touch `process.env.DATABASE_URL`** must `loadEnv()` first, then dynamic-import the db client *inside* `main()` — see `scripts/seed-pirate-class.ts` and `scripts/recompile-pirate-class.ts`.
6. **Branch protection on `main`** is enforced (PR + CI required, admin bypass). Always work on a feature branch + PR. Never push to main.
7. **Use SSH for git push** (HTTPS will fail — David's setup).
8. **This is NOT the Next.js you know** — Next.js 16 App Router with renamed primitives. Read `node_modules/next/dist/docs/` before non-trivial work, and heed deprecation notices.

## Landmines (things that have bitten us)

- **Shared-pack week access**: rows in `weeks` with `parent_user_id IS NULL` belong to the curriculum pack, not a parent. Use `getPlayableWeekForChild(childId, weekId)` for any access check — NOT `getWeekOwnedBy(weekId, parentId)`. The latter silently 404s for pack weeks.
- **`'use server'` files** (anything in `src/lib/actions/`) can only export async functions. If you need to export an error class or sync helper, put it in `src/lib/errors/` (pure, client-safe) — do NOT import from `@/lib/db/*` in client code, postgres pulls in `fs`/`net`/`tls`.
- **`ZodiacIconDefs`** must be mounted in an ancestor of any `<ZodiacIcon>` use. Currently mounted in `src/app/play/[childId]/layout.tsx`.
- **React 19 strict mode** double-renders. Any `onComplete?.()` in a render body double-fires — wrap in `useEffect`.
- **`prefers-reduced-motion`** — every fx component must respect `useReducedMotion()` and fall back gracefully.
- **`DATABASE_URL` shared across Vercel envs** (Neon free tier) — production scripts can write to prod inadvertently. Confirm before running anything that mutates.
- **Bilingual rule for collectibles** (locked): Yinuo is English-native. All pack items render both `nameZh` AND `nameEn` side-by-side. Lore is also dual `loreZh`/`loreEn`. No language toggle. New packs must seed both columns.
- **Per-pack gacha cost lives in `packRegistry.ts`** (`getPackMeta(slug).paidPullCost`), NOT a DB column. Adding a new pack? Add an entry to `PACK_REGISTRY` with `paidPullCost`, `ItemCard` component, `themeEmoji`, bilingual display names + slogan, else `pullPaid` throws "no UI meta" and the per-pack page can't render.
- **`AvatarRender` uses `useId()` for clipPath IDs** — multiple instances on the same page collide otherwise. If you build a similar SVG component, use `useId()` not a hardcoded ID.
- **Never pass `PackUiMeta` (or any function-bearing object) from a server component into a `'use client'` component**: `meta.ItemCard` is a React component and `meta.resolveRevealEmoji` is a callback. RSC silently serialises everything else fine, then crashes at request time with `"Functions cannot be passed directly to Client Components"`. Local tests + `pnpm build` will NOT catch this — only prod / `pnpm dev` exercises the boundary. Fix pattern: take `packSlug: string` instead, and `getPackMeta(slug)` inside the client component. See `PackPageBody.tsx`.
- **`currentThemeSlug` in `src/lib/audio/play.ts` is a module-level singleton** — safe today because there is only one child session per browser tab, and `SoundThemeBootstrap` re-hydrates it on every play-layout mount. If multi-child switching is ever added (e.g. a parent previewing both children's islands in the same tab), the singleton will carry over the previous child's theme until the next mount. Fix at that time by moving state into a React context.
- **Shop purchase dispatch must include all purchasable kinds.** `purchaseShopItemInTx` was avatar-only from PR #21 through PR #33 — pets and sound themes shipped UI that called the action but it threw `ItemNotPurchasableError` at the SQL boundary, silently breaking those tabs in prod. PR #34 added a `switch (kind)` dispatch (`avatar` → inventory side-effect; `sound_theme | pet | decor` → coin debit + `shop_purchases` row only). When adding a new shop kind, extend the switch — don't add a parallel purchase action, or you'll fragment coin-debit and trophy-grant logic. Avatar regression coverage lives in `tests/unit/shop-db.test.ts`.
- **Decor anchors live in TS, not DB.** `src/lib/decor/anchors.ts` is the source of truth for positions; the DB stores only the anchor slug, and `src/lib/decor/catalog.tsx` resolves slug → coords + SVG component. To move a decoration, edit `anchors.ts` and redeploy — no migration needed. To add a new decoration: add an entry to `DECOR_CATALOG` + `ANCHORS`, create the SVG component under `src/components/play/decorations/`, extend `seed-decorations.ts`. The catalog test (`tests/unit/decor-catalog.test.ts`) enforces that every catalog entry has a known anchor and that anchors are distinct.
- **`pinyin_pick` is retired, not deleted.** The React component (`src/components/scenes/PinyinPickScene.tsx`), config schema (`PinyinPickConfigSchema`), and `scene_templates` row all still exist; the template row has `is_active=false`. Old `scene_attempts` rows reference the template via foreign key — don't delete it. Boss `questionTypes` array no longer includes `'pinyin_pick'` (PR #35 narrowed 6 → 5 types). If you ever re-add pinyin practice, you also need to flip `is_active=true` and update `compile-week.ts`.
- **Stable level keys on `week_levels`.** `compileWeekIntoLevels` upserts by `(week_id, level_key)` instead of delete-then-insert. The key shape lives in `compile-week.ts`: `review:flashcard:<characterId>` for per-char review scenes; `practice:<sceneType>:<slotIndex>` for slot-based practice scenes (e.g. `practice:audio_pick:0`); `boss:boss:0` for the boss. When changing the key shape you'll regress all of Yinuo's existing `scene_attempts.week_level_id` linkages unless you write a one-off attempt-migration script. Don't change keys casually.

The section page (`/play/[childId]/level/[weekId]/[section]/page.tsx`) mounts it. **If you add a new "long session" route** (e.g. a tutorial flow), mount `<MidSceneFlag />` inside or tab taps will navigate away without the confirm.

- **Boss-clear card grants use `sessionId` as `refId`, not `weekId`.** PR #52 wired `pullCardForChild('boss_clear', parsed.sessionId)` from `finishLevelAction`. Each boss clear runs in its own `play_sessions` row; `sessionId` is naturally unique per clear so repeat-clear grants work via the `card_grants_log` PK collision dedupe. If you use `weekId` by mistake, only the first ever clear grants a card and every repeat silently no-ops. The `pullCardForChild` is called unconditionally on `bossCleared` (NOT gated on `!alreadyAwarded`) — that gate only protects the coin award, not the card grant.
- **Shards are per-pack, not global.** `shard_balances (childId, packId, shards)` was added pre-PR-#52 and PR #52 starts using it. Dupes of flag cards only buy flag cards; dupes of zodiac only buy zodiac. Don't try to combine balances into a "wallet" UI — the per-pack identity is intentional (Yinuo's strategic choice is which pack to grind).
- **Weekly card cap is 10 ALL-SOURCES per UTC week.** `child_card_grants_weekly (childId, weekStartUtc, count)` tracks the union of boss/perfect_week/story grants. Cap check happens inside `pullCardInTx` under `SELECT...FOR UPDATE` + `onConflictDoUpdate` counter (race-safe). Don't add a new card source without consulting the same counter, or you'll exceed the cap silently. `WEEKLY_CARD_CAP = 10` lives in `src/lib/db/grants.ts`.
- **PR #51's `week_progress.freePullClaimed` column is dead read-side post-PR-#52 but kept in the DB.** PR #51 added it; PR #52 reverted the client gate (chest now shows on every boss clear) but didn't drop the column per the additive-only migration rule. The `finishLevelAction` return still includes `freePullClaimed` for backwards compat; the client just ignores it. Future PRs may reuse the column for "first-ever boss clear celebration" UX.
- **Drizzle migrations DO NOT auto-apply on Vercel deploy by default.** Discovered 2026-06-02: PR #48 (migration 0017 / `story_chapters`) merged + deployed but the table was never created in prod. Every `/play/[childId]` request 500'd with `relation "story_chapters" does not exist`. Tests mock `@/db` so CI couldn't catch it. Earlier CLAUDE.md (and PR #52 spec) incorrectly claimed "auto-applies on Vercel deploy" — this was wrong. **Fix landed 2026-06-02**: `package.json` build script now reads `tsx scripts/migrate.ts && next build`, so every Vercel build applies pending migrations BEFORE the deploy goes live. Drizzle's migrator is idempotent (reads `__drizzle_migrations` journal) so reruns are safe. If a build fails on migrate, the deploy is blocked — that's the desired behavior (better than silently shipping broken code). The migrate script seeds 4 scene_templates rows at the end (`pinyin_pick`, `translate_pick`, `sentence_cloze`, `image_word`) which is also idempotent. Manual `pnpm db:migrate` against prod is still available for emergencies.
- **`pullPaid` action + `GachaPullButton` are deprecated, not deleted.** PR #52 marked both `@deprecated`. The action is still callable (for emergency rollback) but has no UI entry point — `PackPageBody` no longer renders the button. Drop both entirely in PR #53+ once PR #52 is stable in prod.
- **`pullCardForChild` is the trust-caller variant; `swapShardsForItem` calls `requireChild`.** The asymmetry is intentional: `pullCardForChild` is invoked from server actions that already auth-gated (`finishLevelAction`, `markChapterReadAction`); `swapShardsForItem` is a user-tappable action that has to validate. Don't add `requireChild` to `pullCardForChild` — it's not just dead code, it'd add latency to the most performance-sensitive path (boss-clear). And don't drop it from `swapShardsForItem` — anyone could call the action otherwise.
- **`weightedRandomPick` throws on zero-total catalog instead of silently picking.** PR #52 spec reviewer caught: the original code's "degenerate fallback" was actually reachable only when EVERY catalog item had `dropWeight=0` (i.e. an entirely retired catalog), not when "all packs are complete". The throw is the safe behavior — silently picking a zero-weight (retired) item would defeat the retirement. If a future PR retires individual items, filter `dropWeight > 0` at query time before calling this function.
- **`perfect_week` card grant is fire-and-forget with `.catch()`; boss-clear awaits.** PR #52 wired both inside `finishLevelAction`. Boss-clear is the kid's expected reward — `LevelFanfare` renders the card banner so it must await. `perfect_week` is a bonus side-effect; its slow tx shouldn't block the response. The `.catch()` is required — without it, an unhandled promise rejection would surface as Node noise. If you wire a new card-grant call site, decide: latency-critical → await; bonus → `.catch()`.
- **Lianliankan path-finding endpoints are tiles, not empty.** PR #57's `findPath` BFS allows `from` and `to` to contain tiles (they're the matchees) but every intermediate cell MUST be empty. Common refactoring bug: tightening the "must be empty" check to include endpoints, which breaks every match. Test cases `'treats endpoints as tiles'` and `'returns null when intermediate cell has any tile'` guard against this — don't relax their phrasing without thinking. BFS state key is `(row, col, dir, bends)` on all 4 components; visited set keyed on the full tuple so a cell can be revisited from a different direction or with fewer bends already used.
- **`word_match` template stays in DB after PR #57.** Same retirement pattern as `pinyin_pick` (PR #35) and `visual_pick` (PR #51). `scripts/retire-word-match.ts` flips `scene_templates.is_active=false`; `compile-week.ts` no longer emits `word_match` in sight slot 1 (lianliankan replaced it); but `WordMatchScene.tsx`, `WordMatchConfigSchema`, and `SceneRunner case 'word_match'` ALL stay for backwards-compat with old `scene_attempts` rows that reference the template. If you ever re-enable, flip `is_active=true` AND re-add slot logic.
- **Lianliankan board has a 1-cell empty border by design.** The inner 4×2 (8 cells = 4 pairs) is wrapped by an outer ring of empty cells so paths can route around remaining tiles. Don't "optimize" by dropping the border — most Z-bend paths become impossible and the game devolves into trivial straight-line matching. The border is fundamental to the puzzle.
- **Lianliankan blocked-path matches deselect with shake; they do NOT trigger reshuffle.** PR #57 spec §4.3 was explicit: matching pair with no valid ≤2-bend path → shake + deselect, NO penalty, NO reshuffle. Reshuffle is reserved for the actual deadlock case (no valid pair-with-path remaining anywhere) handled by a separate `useEffect`. The spec reviewer caught an implementer who initially deviated — silent board mutation on tap is confusing for a 6yo and removes the puzzle information the kid had built up. Keep the two cases distinct.
