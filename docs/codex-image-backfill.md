## 1. Inputs ‚Äî what to generate

Read prompts from the production Neon Postgres. There are 422 eligible rows. Each row is one `words` record with a non-null `image_hook` and a null `image_url`.

```sql

FROM words
WHERE image_url IS NULL
  AND image_hook IS NOT NULL;
```

## 2. Style preamble ‚Äî must prepend, do not modify

Every prompt sent to the image model must be:

```
cartoon illustration for children, bright colors, simple, single subject, no text: {image_hook}
```

Exact prefix, including the trailing colon + space. This keeps the new 422 visually consistent with the 4 proof-of-life images already in prod. **Do not** add extra style language ("Pixar style", "watercolor", "high detail"); we tested those and they look out of place next to the existing four.

Source of truth for the preamble is `src/lib/ai/pollinations.ts` constant `POLLINATIONS_STYLE_PREAMBLE`. If you change it, update that constant too ‚Äî and be aware you'd then need to regen all 4 existing images for consistency.

## 4. Output contract ‚Äî Vercel Blob

Each generated image must be uploaded to Vercel Blob at a deterministic path:

```
words/{wordId}.png
```

Use `addRandomSuffix: false` and `allowOverwrite: true` so the URL is stable (re-runs overwrite in place). Public access.

```ts
import { put } from '@vercel/blob';

const blob = await put(`words/${wordId}.png`, pngBytes, {
  access: 'public',
  contentType: 'image/png',
  addRandomSuffix: false,
  allowOverwrite: true,
});
// blob.url is the public URL to persist to the DB
```

The Blob store is already public (created 2026-05-28). `.env.local` has both `BLOB_READ_WRITE_TOKEN` and `BLOB_STORE_ID`. Reference store: `mfl7ap4djy0w98ey.public.blob.vercel-storage.com`.

**Landmine:** Vercel Blob distinguishes public vs private *at store creation*. The current store IS public ‚Äî do not flip it. If for some reason you need to recreate, you must select public at creation; the dashboard toggle on an existing private store does not reliably work.

---

## 5. Database write ‚Äî idempotent UPDATE

After upload succeeds, write the public URL back to Postgres, guarded by `image_url IS NULL` so concurrent runs can't double-write.

```sql
UPDATE words
SET image_url = $blobUrl
WHERE id = $wordId
  AND image_url IS NULL;
```

If 0 rows affected, someone else already populated this word ‚Äî that's fine, skip and move on. Do not delete the just-uploaded Blob; it'll just be orphaned-but-overwritable next time.

## 6. Provider notes (you decide)

We've already tried two providers and learned the constraints:

**Pollinations.ai (free, key-less) ‚Äî broken for batch.** Their free tier enforces "1 request queued per IP", and their server-side queue holds a slot indefinitely if your first request hangs past your client timeout. We measured: 0/422 successful across multiple attempts at concurrency=1. **Do not** use the free tier for batch. Paid sub ($5/mo at `enter.pollinations.ai`) lifts the queue limit.

**Gemini API free tier ‚Äî zero quota for image gen.** The plain `generativelanguage.googleapis.com` endpoint returns 429 RESOURCE_EXHAUSTED for `gemini-2.5-flash-image`. Enabling billing on the project at `aistudio.google.com/usage` unlocks pay-as-you-go at ~$0.039/image (‚âà $16.50 for 422). David's consumer Google AI Pro subscription does NOT extend to API access.

**Other viable providers** (untested by us): OpenAI `gpt-image-1` (~$0.011 low / $0.042 medium per image, requires API credits separately from ChatGPT Plus); Vertex AI Imagen (~$0.02/image, requires GCP billing + service account); Replicate / fal.ai / Together (per-image pricing varies, often $0.003-$0.01).

Pick what makes sense for your environment. The contract above (prompt format, Blob path, DB column) is what matters.

## 8. Reference implementation (provider-swappable)

`src/lib/ai/pollinations.ts` is the existing reference ‚Äî same contract you're implementing, just with a Pollinations URL builder. If you implement a fresh provider, keep the public function signature:

```ts
export async function fetchAndUploadImage(
  imageHook: string,
  wordId: string,
): Promise<string>  // returns the public Blob URL
```

That keeps `src/lib/actions/images.ts` and `scripts/backfill-word-images.ts` unchanged ‚Äî both call `fetchAndUploadImage` and don't care which provider produced the bytes.

**Recommended swap pattern:** create `src/lib/ai/{your-provider}.ts` exporting the same `fetchAndUploadImage` signature, then change the import line in `src/lib/actions/images.ts` and `scripts/backfill-word-images.ts`. Don't delete `pollinations.ts` ‚Äî keep it as a fallback we might use again if the paid sub is enabled later.

The backfill driver lives at `scripts/backfill-word-images.ts` ‚Äî Node script, run with `pnpm tsx scripts/backfill-word-images.ts`. It:
1. Loads `.env.local` (must have `DATABASE_URL` + `BLOB_READ_WRITE_TOKEN`)
2. Selects all `words WHERE image_url IS NULL AND image_hook IS NOT NULL`
3. For each, calls `fetchAndUploadImage(imageHook, wordId)`, then UPDATEs with the idempotency guard
4. Prints a `.` per success and `FAIL {text}: {error}` per failure
5. Exits with final tally

You can reuse this driver as-is. The only thing that needs to change for a new provider is the imported `fetchAndUploadImage`.
